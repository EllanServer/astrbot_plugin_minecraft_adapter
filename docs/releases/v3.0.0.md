# MineSentinel v3.0.0

MineSentinel 3.0.0 将项目从“日志分析后给出一份长报告”升级为完整的 Minecraft 事件管理报告系统。新版本更适合直接投递到管理群：先用白话说明发生了什么、涉及谁、需要何时处理，再为每个事件给出可验证的处置方案和原始证据。

## 重点更新

- 默认报告改为“1 页管理摘要 + N 页事件处置单 + 完整日志附件”。
- 不再显示 `P0/P1/P2/P3`，统一使用“马上处理、今天处理、留意观察”等无需培训即可理解的表达。
- AI Provider 可以输出完整的结构化处置方案，而不仅是一句笼统建议。
- 报告会明确列出时间、地点、人物、插件或服务、日志文件五类事实；没有证据时明确写“未关联具体玩家”或“未识别具体插件”，不会编造。
- 新增事件生命周期、稳定事件编号、恢复跟踪和复盘报告。
- 使用真实公开日志完成回归，修复普通聊天误报、玩家名遗漏和重复事件编号等问题。

## 全新事件管理报告

默认 `report.layout=incident_management`。一份报告现在包含：

1. **管理摘要**：报告范围、当前状态、新发/持续/已恢复数量、最需要先做的三件事。
2. **事件处置单**：每个活动事件单独一页，避免多个无关故障挤在同一张长图中。
3. **完整日志附件**：保留 `JSONL` 或 `JSONL.GZ` 原始证据，正文只展示最关键的日志行。

每张处置单固定展示：

- 准确发生时间和持续时长；
- 服务器、后端服、世界和坐标等位置；
- 有证据关联的具体玩家；
- 有证据关联的插件、数据库或外部服务；
- 原始日志文件名；
- 影响判断、当前结论、负责人和完成标准；
- 按顺序排列的检查与解决步骤。

图片生成失败时会自动回退到同结构文本，不会丢失处置内容。

## 面向普通管理人员的表达

报告不再要求读者理解运维分级代码：

| 旧表达 | 新表达 | 建议开始时间 |
| --- | --- | --- |
| P0/P1 | 马上处理 | 现在开始 |
| P2 | 今天处理 | 今天内 |
| P3 | 留意观察 | 下次巡检时 |

技术命令和配置项被下沉到“给维护人员的详细步骤”。管理摘要只回答四个问题：发生了什么、谁负责、什么时候处理、怎样算处理完成。

## AI Provider 结构化处置方案

事件级 AI 诊断协议新增 `check_plan`。AI 参与诊断时必须返回 2 至 6 个有顺序的步骤，每步都包含：

- `phase`：当前阶段；
- `check`：具体检查或处理动作；
- `expected`：通过标准；
- `on_failure`：没有通过时的下一步。

最后一步必须是恢复验证或完成确认。通过校验后，事件处置单会直接使用 AI 方案，并标记“AI 给出的详细步骤”。

为防止 AI 越权或污染事实，新版本继续锁定以下字段：

- 事件分类和处理等级；
- 时间、服务器、世界和坐标；
- 玩家、插件和日志文件；
- 证据数量和原始证据索引；
- 事件集合及其生命周期。

AI 方案字段缺失、包含危险操作、与事件证据不匹配或缺少恢复验证时，会自动回退到确定性规则方案。AI 仍可在受限范围内扩展命中点前后日志，或按配置通过 AstrBot 网页搜索查询官方资料。

## 事件生命周期与复盘

- 周期巡检会跟踪 `new`、`ongoing`、`recovered` 三种状态。
- 恢复事件只在首次退出活动集后的一个周期中出现，避免重复占用摘要。
- 新事件编号采用稳定的 12 位摘要，例如 `MS-AEA9977EBCA3`。
- 同类问题在多个独立时间段重复出现时会获得不同编号，不再出现编号冲突。
- 新增 `/ms report postmortem [server] [24h]`，可按复盘口径生成同一套证据化处置页。
- 生命周期文件只保存事件摘要，不保存原始聊天或日志正文。

## 更准确的人物与插件识别

玩家名现在可从以下证据提取：

- Minecraft 聊天记录；
- Vulcan 等反作弊结构字段；
- `playerName`、`targetPlayer`、`actor`、`sender` 等上下文；
- `moved wrongly`、`moved too quickly`、登录、退出和命令记录等原生服务端格式。

插件与组件名称可从以下位置识别：

- 结构化插件字段；
- 日志前缀和调度线程名；
- Hikari/连接池名称；
- `plugins/<插件>/...` 配置路径；
- 错误证据中的插件标识。

系统只会把同一事件证据中出现的人物与插件写入处置单。玩家只是在线、但没有证据参与某个数据库或插件故障时，报告会写“未关联具体玩家”，不会为了填满字段而错误关联。

## 故障域拆分与具体方案

技术事件会按根因边界拆分，不再因为发生在同一服务器或相邻几分钟就合成一个“大事件”。当前独立处理：

- 插件配置和内容解析；
- 数据库、连接池和存储；
- 经济、商店与资产流水；
- 外部 API、翻译和资源服务；
- 网络、代理和跨服链路；
- 身份认证与后端访问边界；
- TPS/MSPT、GC、内存和磁盘性能；
- 传送、位置、世界与区块；
- 玩家反馈、权限问题和社区管理。

数据库方案会覆盖只读连通测试、连接池配置、数据库上限和恢复验证；经济方案会区分交易主链路与可选资源服务，避免把翻译服务超时误判为商店交易故障。

## 误报和真实性修复

- “你去 tp”“我想传送”“太远了传送就好”等普通聊天不再升级为传送异常。
- `/home 后卡住`、掉入虚空、位置错误、无法传送等带明确异常语义的反馈仍会进入处置流程。
- 启动/关闭阶段的 Hikari、AstrbotAdapter、CMI 和正常代理握手继续作为生命周期噪声过滤。
- `Unknown or incomplete command` 等无充分故障语义的内容不会单独制造严重事件。
- 同一故障的重复堆栈会聚合，首条根因不会被刷屏淹没。
- QuickShop 外部 Crowdin 服务超时不会再错误套用数据库 `SELECT 1` 方案。

## 图片与投递升级

- 新增多页报告渲染和多图顺序发送。
- 首图保持兼容，旧调用仍可读取 `.image`；新调用使用 `.images` 获取全部页面。
- 图片全部成功后再发送日志附件；任一页失败时回退文本报告。
- 文件发送继续使用 AstrBot `File` 组件，支持完整 `JSONL/JSONL.GZ` 证据归档。
- 摘要页限制行动项和事件索引数量，超出部分明确给出省略数量并指向完整报告。

## 配置与兼容性

新增或正式启用的报告配置包括：

- `report.layout=incident_management`；
- `report.lifecycle_enabled=true`；
- `report.max_summary_incidents`；
- `report.max_detail_pages`。

兼容性说明：

- 内部插件 ID 和安装目录继续使用 `astrbot_plugin_minecraft_adapter`，已有 AstrBot 安装无需改目录。
- 数据目录继续使用 `plugin_data/mine_sentinel`，旧路径迁移逻辑保留。
- 旧 `overall`、`incidents`、`community`、`player_problems`、`risk_actions` 五段数据保留在 `legacy_sections`。
- 设置 `report.layout=legacy` 可继续使用旧图片布局。
- 新结构化报告使用 `incident_management.schema_version=3`。
- 面向新结构的自动化消费者应读取 `action_label`、`timing` 和 `check_plan_source`，不要再依赖 P 级代码。

## 升级注意事项

1. 升级前备份插件目录、AstrBot 配置和 `plugin_data/mine_sentinel`。
2. 使用本 Release 的插件 ZIP 替换代码，不要把旧版已经删除的文件混回新目录。
3. 保留原有配置和数据目录，启动后检查配置迁移日志。
4. 执行 `/ms monitor status` 确认日志源和投递目标。
5. 执行 `/ms report now <服务器ID> 30m` 验证摘要、处置单和附件投递。
6. 使用 Rust 加速时，只安装本 Release 中与操作系统和 CPython 版本完全匹配的 wheel；不匹配时保持纯 Python 模式即可。

README 已提供可直接交给本机 AI 助手的完整安装提示词和升级提示词。
